# 个人工具平台 -- 开发规范与经验总结

> 适用环境：Python 3.x + Flask + SQLite，部署于 PythonAnywhere 免费云服务。  
> 目的：避免重复踩坑，保证新模块开发一致性，确保部署流程可靠。

---

## 一、项目架构总览

```
app.py                     # Flask 主入口：认证、路由注册、启动初始化
├─ routes/utils.py         # 共用工具函数（数据库、日志、时区、加密）
├─ routes/deploy.py        # 部署工具 （git-pull、状态监控）
├─ routes/backup.py        # 自动备份清理
├─ routes/renqing.py       # 人情礼金系统
├─ routes/paiban.py        # 排工考勤系统
├─ routes/gpa.py           # GPA成绩系统
├─ routes/hsgrades.py      # 高中成绩系统
├─ routes/countdown.py     # 倒计时系统（公历/农历）
├─ routes/accounting.py    # 个人记账系统（收支/借贷管理）
├─ routes/pa.py            # PA 自动续期
└─ routes/seed_data.py     # 种子数据库（base64编码，自动生成）

各模块前端 HTML 存放在对应中文目录下：
人情/  排班/  绩点/  成绩/  倒计时/  记账/  部署/
```

**核心原则**：
- 每个业务模块 = 1 个 Blueprint + 1 个 HTML 前端 + 1 个 SQLite 数据库
- 模块之间完全独立，不相互导入
- 前端为纯原生 HTML/CSS/JS（无框架、无构建工具）
- 部署 API 优于一切业务模块，确保即使某模块崩溃也能远程恢复

## 二、代码规范

### 2.1 Python 缩进

**必须使用 4 个空格缩进，严禁使用 Tab 或混用。**

这是本项目最惨痛的教训：`countdown.py` 中 `create_event()` 函数体无意缩进了 8 个空格，导致 `IndentationError`，整个 Flask 应用崩溃，部署 API 也一起挂掉，形成死锁。

**预防措施**：
- 编写代码后立即执行 `python -m py_compile xxx.py` 检查语法
- 提交前运行 `python -c "from routes.xxx import bp; print('OK')"` 验证导入
- 使用编辑器的「显示空白字符」功能检查缩进

### 2.2 文件头声明

每个 `.py` 文件必须包含：

```python
# -*- coding: utf-8 -*-
```

### 2.3 导入顺序

```python
# 1. 标准库
import os, json, time
from datetime import datetime

# 2. 第三方库
from flask import Blueprint, jsonify, request

# 3. 项目内导入（相对导入）
from .utils import TZ, make_logger, make_db
```

### 2.4 异常处理

**每个 API 路由必须用 try/except 包裹，返回统一 JSON 格式：**

```python
@bp.route('/api/xxx/action', methods=['POST'])
def action():
    try:
        # 业务逻辑
        return jsonify({'success': True, 'data': result})
    except Exception as e:
        return jsonify({'success': False, 'error': str(e)}), 500
```

### 2.5 日志规范

使用 `utils.make_logger()` 创建模块级日志：

```python
from .utils import make_logger
_log = make_logger(os.path.join(BASE_DIR, '目录', '模块名.log'))

def _log(msg):
    """简单日志，带时间戳"""
    # make_logger 自动处理目录创建和时间戳
```

### 2.6 敏感配置管理

**Token、密钥、密码等敏感值禁止硬编码在代码中。** 统一使用环境变量 + 默认值兜底模式：

```python
# 正确写法
DEPLOY_TOKEN = os.environ.get('DEPLOY_TOKEN', '默认开发用值')
```

- 代码泄露时 Token 不会暴露
- 生产和开发环境可用不同的 Token
- 如果环境变量未设置，自动降级到默认值，不影响开发调试

## 三、新增模块开发模板

### 3.1 创建后端路由文件 `routes/new_module.py`

```python
# -*- coding: utf-8 -*-
"""新模块说明"""
import os, json

from flask import Blueprint, jsonify, request

from .utils import TZ, make_logger, make_db

bp = Blueprint('new_module', __name__)
BASE_DIR = os.path.dirname(os.path.dirname(os.path.abspath(__file__)))

# ========== 数据库 ==========
DB_DIR = os.path.join(BASE_DIR, '新模块目录')
DB_FILE = os.path.join(DB_DIR, 'data.db')
_get_db = make_db(DB_FILE)

def init_db():
    """初始化数据库表"""
    os.makedirs(DB_DIR, exist_ok=True)
    conn = _get_db()
    conn.execute('''CREATE TABLE IF NOT EXISTS xxx (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        ...,
        updated_at TEXT DEFAULT (datetime('now','localtime'))
    )''')
    conn.commit()
    conn.close()

# ========== 日志 ==========
_log = make_logger(os.path.join(DB_DIR, 'app.log'))

# ========== API 路由 ==========

@bp.route('/api/new_module/data', methods=['GET'])
def get_data():
    """读取数据"""
    try:
        conn = _get_db()
        rows = conn.execute('SELECT * FROM xxx ORDER BY id').fetchall()
        conn.close()
        return jsonify({'success': True, 'data': [dict(r) for r in rows]})
    except Exception as e:
        return jsonify({'success': False, 'error': str(e)}), 500


@bp.route('/api/new_module/data', methods=['POST'])
def save_data():
    """保存数据"""
    try:
        data = request.get_json(silent=True) or {}
        # 参数校验
        if not data.get('name'):
            return jsonify({'success': False, 'error': '缺少必要参数'}), 400
        # 业务处理
        conn = _get_db()
        conn.execute('INSERT INTO xxx (name, ...) VALUES (?, ...)', (data['name'], ...))
        conn.commit()
        conn.close()
        _log(f'新增记录: {data["name"]}')
        return jsonify({'success': True})
    except Exception as e:
        return jsonify({'success': False, 'error': str(e)}), 500
```

### 4.1 强制使用 `make_db()` 工厂

**严禁在任何模块中直接 `sqlite3.connect()`。** 必须统一通过 `utils.make_db()` 创建连接工厂。

```python
from .utils import make_db
_get_db = make_db(db_file_path)
```

**禁止的写法**（以下两种都会导致问题）：

```python
# 禁止：直接导入 sqlite3
import sqlite3
conn = sqlite3.connect(path)

# 禁止：手动设置 PRAGMA
conn = sqlite3.connect(path)
conn.execute('PRAGMA journal_mode=WAL')
conn.execute('PRAGMA busy_timeout=5000')
```

**原因**：直接 `sqlite3.connect()` 缺少 WAL 日志模式（并发读写保护）和 busy_timeout（锁等待），在高并发或异常情况下会导致 `database is locked` 错误。本项目曾因 gpa.py 和 hsgrades.py 绕过 make_db() 而埋下隐患，已在 2026-07-19 审查中修复。

`make_db()` 自动设置：
- `PRAGMA journal_mode=WAL` -- 写前日志，支持并发读
- `PRAGMA synchronous=NORMAL` -- 平衡安全和性能
- `PRAGMA busy_timeout=5000` -- 5 秒忙等待超时

### 4.2 数据库位置

- 存放在模块对应的中文目录下（如 `人情/gifts.db`）
- 数据库文件不纳入 git（`.gitignore` 中 `*.db`）
- 如有种子数据需求，使用 `gen_seed.py` 生成 base64 编码的 `seed_data.py`

### 4.3 表设计规范

- **必须包含**：`id INTEGER PRIMARY KEY AUTOINCREMENT`
- **建议包含**：`created_at TEXT DEFAULT (datetime('now','localtime'))`、`updated_at`
- **JSON 存储**：简单配置类数据可用单行 JSON 存储，减少表结构变更
- **Schema 版本**：如需表结构迁移，添加 `schema_version` 表记录版本号

### 4.4 数据库操作

```python
# 查询
conn = _get_db()
rows = conn.execute('SELECT * FROM xxx WHERE ...', (param,)).fetchall()
conn.close()

# 写入（自动事务）
conn = _get_db()
conn.execute('INSERT INTO xxx (...) VALUES (?, ?, ...)', (a, b, c))
conn.commit()
conn.close()

# 更新
conn = _get_db()
conn.execute('UPDATE xxx SET field=? WHERE id=?', (value, eid))
conn.commit()
conn.close()
```

### 4.5 SQL 拼接安全规则

**用户输入值必须用参数化查询（`?` 占位符），禁止 f-string 拼接用户数据。**

| 场景 | 做法 | 示例 |
|---|---|---|
| 用户输入值 | 必须 `?` 占位符 | `conn.execute('WHERE name=?', (user_input,))` |
| 内部构造的表名 | 允许 f-string | `f'SELECT * FROM [{table_name}]'`（表名来自 DB 元数据） |
| 内部构造的子句 | 允许 f-string | `w = 'WHERE ...' if condition else ''`（子句逻辑在代码内） |
| LIKE 查询 | 参数化 + 拼接 | `conn.execute('WHERE name LIKE ?', (f'%{kw}%',))` |

**关键原则**：只要数据来源不是用户直接输入，而是代码内部控制的值（表名从 `sqlite_master` 查询、子句由 if/else 决定），则 f-string 是安全的。如有疑问，一律用 `?` 占位符。

```python
# 正确：用户输入用占位符
conn.execute('SELECT * FROM records WHERE name LIKE ?', (f'%{keyword}%',))

# 正确：表名来自 DB 元数据，使用 f-string
for (table_name,) in conn.execute("SELECT name FROM sqlite_master WHERE type='table'"):
    conn.execute(f'SELECT COUNT(*) FROM [{table_name}]')

# 错误：用户输入直接拼进 SQL（SQL 注入风险！）
conn.execute(f"SELECT * FROM records WHERE name = '{user_name}'")
```

### 5.2 设计原则

- **纯文字版本**：不使用图标/emoji 符号，状态和操作用文字表达
- **颜色方案**：CSS 变量统一管理，支持深色/浅色切换
- **SPA 单页**：前端状态用 JS 变量管理，不依赖 URL hash

### 5.3 移动端适配（强制）

本项目多数场景为手机端使用，所有前端页面必须适配移动端。

**核心原则：双模式设计 —— 移动端 App 效果 + PC 端大屏显示**

| 模式 | 断点 | 布局 | 导航 | 内容宽度 |
|------|------|------|------|----------|
| 移动端 App | ≤768px | 全屏沉浸式 | 底部固定导航栏 | 100vw，padding 12-16px |
| PC 端大屏 | >768px | 传统桌面布局 | 顶部/侧边导航栏 | max-width 限制，居中显示 |

通过 `@media (max-width: 768px)` 和 `@media (min-width: 769px)` 媒体查询实现两种模式切换。**统一断点为 768px**，不允许使用其他断点值（如 640px、480px）。

**移动端 App 效果要求（≤768px）**：
- **全屏沉浸布局**：`html, body { height: 100%; overflow: hidden }`，body 使用 `display: flex; flex-direction: column` 占满视口
- **底部固定导航**：使用 `position: fixed; bottom: 0` 的底部标签栏切换页面，**仅在移动端显示**（`@media (min-width: 769px) { .bottom-nav { display: none; } }`）
- **滑动操作**：支持滑动删除等手势交互
- **下拉刷新**：支持下拉刷新数据
- **弹窗全屏**：编辑/新增弹窗在移动端全屏显示

**PC 端大屏要求（>768px）**：
- **内容居中**：主要内容区域设置 `max-width`（建议 900-1400px），`margin: 0 auto` 居中
- **顶部导航**：使用页面顶部的导航标签/按钮，**隐藏底部导航栏**
- **充分利用宽度**：统计面板、表格等使用多列网格或自适应 `auto-fit` 布局
- **侧边栏可选**：管理类页面可使用侧边栏布局（如部署页）

**布局**：
- 使用单列流体布局，避免多列并排（统计图表除外）
- 页面最大宽度不超过 100vw，禁止水平滚动
- 卡片/表单容器使用 `width: 100%` + `max-width` 限制，左右留适当 padding（建议 12px-16px）

**触控**：
- 所有可点击元素（按钮、链接、选择器）最小触控区域 44x44px
- 相邻可点击元素间距至少 8px，防止误触
- 表单输入框高度不小于 40px

**字体**：
- 正文字号不小于 14px（iOS 会自动缩放小于 16px 的输入框），输入框字号建议 16px
- 数字/金额展示字号适当加大（18px-22px），便于快速扫读

**Viewport**：
```html
<meta name="viewport" content="width=device-width, initial-scale=1.0, viewport-fit=cover">
```

**安全区域**（刘海屏/底部指示条）：
```css
body {
    padding-top: env(safe-area-inset-top);
    padding-bottom: env(safe-area-inset-bottom);
    padding-left: env(safe-area-inset-left);
    padding-right: env(safe-area-inset-right);
}
```

**表格/列表**：
- 表格列数超过 3 列时，考虑卡片式布局替代表格
- 必须用表格时，设置 `overflow-x: auto` 允许横向滚动
- 长文本使用 `overflow: hidden; text-overflow: ellipsis; white-space: nowrap`

**弹窗/对话框**：
- 弹窗宽度使用 `min(90vw, 500px)`，确保小屏不溢出
- 弹窗关闭按钮放在显眼位置（右上角或底部），触控区域足够大

**测试要求**：
- 开发完成后用 Chrome DevTools 模拟 iPhone SE（375px）、iPhone 12（390px）、iPad（768px）三种宽度测试
- 检查所有表单输入、按钮点击、下拉选择在触控模式下的可用性

## 六、部署流程规范

### 6.1 部署流水线

```
本地修改代码 → git commit → git push → 触发部署 API → 服务器 git pull → 自动重载
```

### 6.2 每次代码修改后的必须步骤

1. **语法检查**：`python -m py_compile 修改的文件.py`
2. **导入测试**：`python -c "from routes.xxx import bp; print('OK')"`
3. **本地运行测试**：`python -c "from app import app; c=app.test_client(); ..."`
4. **提交推送+部署（原子操作，不可拆分）**：
   ```
   git add ... && git commit -m "..." && git push origin master:master master:main --force
   curl -X POST https://slhfwq.pythonanywhere.com/api/git-pull?force=1 -H "X-Deploy-Token: ce952b9ded0733ed" -H "Content-Type: application/json"
   ```
   提交推送和触发部署必须连续执行，视为一个原子步骤。部署失败应重试，不能跳过。
5. **验证线上**：`curl https://slhfwq.pythonanywhere.com/api/xxx/data`

### 6.3 部署死锁的教训

**问题**：部署 API 是 Flask 路由，代码有 bug → 应用崩溃 → 部署 API 不可用 → 无法远程恢复。

**解决方案**（已实施）：
- `deploy` 蓝图在 `app.py` 中最先注册
- 所有路由模块用 `_safe_import()` 容错导入，单个模块出错不影响 deploy 和其他模块
- 如果有模块导入失败，会在启动日志中打印 `[WARNING]`

### 6.4 紧急恢复方案

如果应用完全崩溃（部署 API 不可用）：
1. 登录 PythonAnywhere，打开 Bash Console
2. 进入代码目录：`cd /home/slhfwq/construction-log-system`
3. 拉取最新代码：`git pull`
4. 清除缓存：`find . -name "__pycache__" -type d -exec rm -rf {} +`
5. Web 标签页点击 **Reload**

- `os.execv` 会替换当前进程，使 Flask 自动重新加载所有模块
- 部署 API 注册在 `deploy` 蓝图中，该蓝图在 `app.py` 中最先注册

## 七、坑点清单（避雷指南）

### 7.1 缩进错误（已踩坑）
**现象**：XX模块报 500，排查发现缩进是 8 个空格而非 4 个  
**根因**：代码编辑时误按 Tab 或粘贴时缩进错乱  
**预防**：提交前用 `py_compile` 检查每个文件语法；开启编辑器空白字符显示

### 7.2 服务器路径不一致（已踩坑）
**现象**：本地 `~/tools` 操作正常，服务器仍 500  
**根因**：服务器实际代码路径是 `/home/slhfwq/construction-log-system`  
**预防**：永远通过部署 API 触发更新，不要假设服务器路径

### 7.3 中文目录名跨平台（已踩坑）
**现象**：Windows 开发正常，Linux 部署报路径错误  
**根因**：中文目录名编码在不同 OS 下处理不一致  
**对策**：统一使用 `os.path.join()` 拼接路径；所有 `.py` 文件声明 `utf-8` 编码

### 7.4 可选依赖崩溃（已踩坑）
**现象**：倒计时模块因 `zhdate` 导入失败而崩溃  
**根因**：生产环境未安装农历库，但代码在模块顶层 `import` 导致 ImportError  
**对策**：可选依赖使用 try/except 懒加载，在函数内部检查可用性。参考 `countdown.py` 中 `HAS_LUNAR` 的处理方式

### 7.5 数据库连接不关闭
**现象**：SQLite 报 `database is locked`  
**根因**：连接使用后未关闭，或异常路径未关连接  
**对策**：每次请求创建新连接，用完立即 `conn.close()`；避免在循环中反复开关连接

### 7.6 部署 API 依赖应用（已踩坑）
**现象**：代码 bug 导致全站 500，部署 API 也无法使用  
**根因**：部署 API 是 Flask 蓝图的一部分，需要应用成功启动  
**对策**：详见第六章「部署死锁的教训」

### 7.7 路径分隔符
**现象**：`routes.randeng\utils` 或 `人情/index.html`  
**根因**：混用 `\` 和 `/` 分隔符  
**对策**：全部使用 `os.path.join()` 和 `os.sep`；URL 路径只用 `/`

### 7.8 PA 磁盘配额
**现象**：文件写入失败  
**根因**：PythonAnywhere 免费账户有严格的空间限制  
**对策**：定期清理日志；数据库备份保留最近 7 个；定期清理 `__pycache__`

### 7.9 Token 硬编码（已修复）
**现象**：`DEPLOY_TOKEN` 写死在代码中  
**风险**：代码推送至公开仓库则 Token 泄露，任何人可触发部署  
**改善**：迁移到 `os.environ.get('DEPLOY_TOKEN', '默认值')`，环境变量优先

### 7.10 绕过 make_db() 直接 sqlite3.connect()（已踩坑/已修复）
**现象**：gpa.py、hsgrades.py 直接 `import sqlite3` + `sqlite3.connect()`  
**根因**：早期模块未统一规范，各自实现数据库连接  
**后果**：缺少 WAL 日志模式和 busy_timeout，并发场景下可能触发锁错误  
**对策**：强制所有模块使用 `make_db()` 工厂，已在 2026-07-19 审查中统一修复

### 7.11 f-string 拼 SQL 的误判
**现象**：`f'SELECT COUNT(*) FROM [{t}]'` 看起来像 SQL 注入  
**实际**：`t` 来自 `sqlite_master` 系统表（DB 元数据），不是用户输入  
**原则**：审查 f-string SQL 时，关键是判断拼接值来源——内部可控 = 安全，用户输入 = 注入风险

### 7.12 JS 中事件状态列表查找遗漏（已踩坑）
**现象**：人情模块中，归档/未归档事件的数据查找逻辑只在 `events`（已归档列表）中 `find`，未引入 `isUnarchivedMode` 判断  
**根因**：`events` 变量存储的是已归档事件子集，未归档事件存在 `unarchivedEvts` 变量中。代码中 4 处 `events.find(e=>e.id===selEvtId)` 未根据 `isUnarchivedMode` 在正确列表中查找  
**影响**：查看未归档事件时，翻页保存拿不到正确的 `page_size` 和 `event_date`；事件详情面板的归档按钮状态不更新  
**修复**：全部改为 `(isUnarchivedMode ? unarchivedEvts : events).find(e=>e.id===selEvtId)`  
**原则**：前端维护多个并行数据子集（如归档/未归档）时，所有查找操作必须根据当前模式选择正确的列表

### 7.13 Response 对象属性混用 .ok 和 .success（已踩坑/已修复）
**现象**：savePage 中用 `r.value.success` 判断 fetch 结果，导致即使后端返回错误（如 400），前端也静默报告"保存成功"  
**根因**：`Promise.allSettled` 返回的每个 `r.value` 是 fetch 的 Response 对象，该对象只有 `.ok` 属性，没有 `.success` 属性。后端 JSON 中的 `success` 字段需要先 `await r.value.json()` 才能访问  
**错误写法**：`results.filter(r=>r.status==='fulfilled'&&r.value.success)` — 永远是 0 条失败  
**正确写法**：`results.filter(r=>r.status==='fulfilled'&&r.value.ok)` — 用 Response 原生属性  
**原则**：fetch Response 对象用 `.ok` 判断 HTTP 状态，后端业务逻辑的 `success` 需先解析 JSON

### 7.14 HTML 行内事件处理器括号不匹配（已踩坑/已修复）
**现象**：number 输入框改动后预览不更新，浏览器控制台报 `Uncaught SyntaxError: missing ) after argument list`  
**根因**：`oninput="updateState('key', parseFloat(this.value)||1">` 中 `updateState(` 调用缺少右括号，HTML 解析后 JavaScript 立即执行报错，整行事件失效  
**修复**：补齐为 `oninput="updateState('key', parseFloat(this.value)||1)"`  
**原则**：行内事件属性本质是 JavaScript 代码片段，必须保持完整语法；使用 IDE/linters 检查 HTML 中的 inline JavaScript；复杂逻辑优先写到 `<script>` 中通过 `addEventListener` 绑定，减少括号嵌套错误
